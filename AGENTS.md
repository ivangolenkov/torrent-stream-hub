# Torrent-Stream-Hub: Руководство для AI-агентов (AGENTS.md)

## О проекте
**Torrent-Stream-Hub** — это гибридный торрент-клиент со стриминг-сервером, разрабатываемый для работы в локальных сетях (особенно на слабых устройствах, таких как NAS).
Основная задача проекта: объединить классический торрент-клиент (фоновое скачивание, сидирование) с возможностями последовательной (Sequential) загрузки файлов для HTTP-стриминга в реальном времени с поддержкой `Range`-запросов.
Сервис эмулирует API TorrServer (для совместимости с существующими Smart TV приложениями вроде Lampa/Num) и предоставляет собственный REST API для Web GUI (Vue 3).

Стек технологий: **Go** (Бэкенд, `anacrolix/torrent`, SQLite WAL) + **Vue 3** (Фронтенд, TypeScript, Vite, Pinia, Tailwind).

## Подход к разработке и взаимодействию

Разработка проекта разбита на строгие последовательные **фазы (этапы)**.

### 1. Планирование задач (Директория `tasks/`)
Весь процесс разработки разделен на 6 фаз. Описание каждой фазы находится в директории `tasks/` в виде отдельных markdown-файлов (например, `phase1-setup-and-core.md`, `phase2-db-and-basic-engine.md` и т.д.).
Каждый файл содержит:
* Краткое описание этапа.
* Конкретный список задач.
* **Definition of Done (DoD)** — чек-лист критериев готовности этапа.

**Правило для агентов:** Строго придерживаться плана. Не переходить к задачам из следующей фазы до тех пор, пока текущая фаза не будет полностью завершена и проверена по DoD.

### 2. Отчетность и контроль (Директория `tasks/implementation/`)
После завершения работы над каждой фазой (или значимой частью фазы), агент **обязан** создать или обновить отчетный файл в папке `tasks/implementation/` (например, `phase1-report.md`).

**В отчете необходимо отразить:**
1. **Что было сделано:** Краткий перечень реализованного функционала и созданных/измененных файлов.
2. **Статус выполнения DoD:** Чек-лист из исходной задачи с отметками о выполнении.
3. **Причины отхода от плана (если применимо):** Если какой-то пункт DoD не был выполнен, или реализация потребовала изменения архитектуры/ТЗ, здесь необходимо детально описать причину такого решения.

### 3. Основные технические ограничения (Важно для агентов!)
* **БД:** Использовать только SQLite. Обязательно применение `PRAGMA journal_mode=WAL;` и настройка пула соединений.
* **Торрент-движок:** Ограничить использование `mmap` в `anacrolix/torrent` для экономии оперативной памяти.
* **QoS (Стриминг):** Реализовать механизм Reference Counting (счетчик активных стримов) и Debounce-таймер при перемотке (Seek), чтобы избегать "мигания" приоритетов при кратковременных обрывах соединения плеером.
* **Безопасность диска:** Обязательно реализовать обработку статуса `DiskFull` при падении свободного места ниже заданного лимита.
* **CORS:** На всех эндпоинтах должен быть разрешен доступ со всех доменов (`*`), так как это критически важно для работы веб-плееров и Smart TV приложений в локальной сети.

### 4. Тестирование (Unit & Integration)
**Правило для агентов:** Необходимо в разумных пределах покрывать тестами все, что реализовывается в рамках плана (каждой фазы).
* Для бизнес-логики и утилит пишем Unit-тесты (`go test`).
* Для слоя работы с БД (repository) пишем интеграционные тесты с использованием временной SQLite базы.
* Все тесты должны успешно проходить (`go test ./...`) перед переходом к следующей фазе.

### 5. Жизненный цикл работы агента
1. Получить команду от пользователя на старт конкретной фазы.
2. Прочитать соответствующий `.md` файл в папке `tasks/`.
3. Составить план действий (через TodoWrite tool, если применимо).
4. Реализовать код (Go / Vue / DB).
5. Написать тесты и проверить код (Линтеры, `go test ./...`, `go build`).
6. Создать отчет в `tasks/implementation/`.
7. Сообщить пользователю о завершении и готовности к следующему этапу.
