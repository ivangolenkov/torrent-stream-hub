# Фаза 18: Оптимизация скорости за NAT — отчет

## Что было сделано
В рамках оптимизации работы `anacrolix/torrent` для клиентов без проброшенных портов были применены следующие меры:

1. **Отключение uTP по умолчанию**:
   - `HUB_BT_DISABLE_UTP` теперь по умолчанию равен `true`. Это обходит проблему, когда библиотека агрессивно сбрасывает TCP-сокеты с хорошими пирами из-за неудачной расшифровки пакетов `ut_holepunch`. Клиент перешел на использование исключительно TCP-соединений.

2. **Повышение лимитов дефолтного профиля**:
   - Профиль `balanced` теперь использует лимиты, аналогичные `aggressive`: `BTEstablishedConns=200`, `BTHalfOpenConns=200`, `BTTotalHalfOpen=2000`, `BTDialRateLimit=200`. Это необходимо для того, чтобы клиент "прощупывал" сеть массированно, компенсируя отсутствие входящих подключений. С учетом новой логики Boost-а, пиковые лимиты соединений при поиске пиров будут составлять 400!

3. **Тонкая настройка таймаутов Engine**:
   - Сохранены очень короткие таймауты установки соединения (`NominalDialTimeout=5s`, `MinDialTimeout=2s`).
   - Увеличен `HandshakesTimeout` до 20 секунд. Это дает клиентам, скрытым за сложным NAT, достаточно времени, чтобы прислать рукопожатие, исключая проблему слишком быстрого сброса уже установленного сокета.

4. **Ускорение CPU-вычислений**:
   - Параметр `PieceHashersPerTorrent` увеличен с дефолтных 2 до 4. Это устранит возможное узкое горлышко при быстрой проверке частей на мощных каналах.

5. **Тестирование**:
   - Были обновлены юнит-тесты в `internal/config/config_test.go` и `internal/engine/config_test.go`, чтобы отражать новые дефолтные значения. Все проверки проходят успешно (`go test ./...`).
