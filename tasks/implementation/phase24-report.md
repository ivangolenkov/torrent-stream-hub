# Phase 24: Engine Refactoring & Best Practices (Отчет)

## Что было сделано

1. **Safety Net (Тестирование):**
   - Написан интеграционный тест `engine_lifecycle_test.go`, который покрывает полный жизненный цикл торрента внутри движка (`AddMagnet` -> `Pause` -> `Resume` -> `Delete`). Этот тест помог выявить несколько серьезных дедлоков (взаимных блокировок), которые ранее скрывались в коде, и успешно предотвратил регрессию логики.

2. **Ампутация переусложненного Swarm Watchdog:**
   - Из движка и API полностью удалена концепция `HardRefresh` (передобавление торрента) и `RecycleClient` (полная перезагрузка клиента `anacrolix`). 
   - Удалены все сопутствующие структуры сохранения стейтов (`hardRefreshSavedState`, логика восстановления метаданных и подсчета попыток). 
   - Взамен оставлен простой и надежный `SoftRefresh` (в рамках обновленного `checkSwarms`), который, в случае застревания скорости скачивания на нуле более 1 минуты, просто просит движок обновить анонсы на трекерах и DHT (без пересоздания клиента).

3. **Гранулярные блокировки (Fine-grained locking):**
   - **Устранение God-Mutex:** Глобальный лок `Engine.mu` (тип `sync.RWMutex`) больше не блокирует весь движок при длительных операциях (таких как `checkSwarms`, `manageResources` или обратные вызовы событий из библиотеки). Теперь он используется исключительно в режиме `RLock` для безопасного чтения мапы `managedTorrents`, а `Lock` применяется только для добавления и удаления торрентов из мапы.
   - Каждому торренту выдан собственный локальный мьютекс `ManagedTorrent.mu sync.Mutex`. Любое изменение статуса, скорости скачивания или прогресса торрента блокирует только его самого.
   - **Устранен классический Deadlock:** Ранее движок `anacrolix/torrent` отправлял события (например, `TrackerConnected`) через функцию `handleStatusEvent`, которая пыталась заблокировать `Engine.mu`. Если в этот момент планировщик ресурсов или Watchdog уже держал `Engine.mu`, приложение намертво зависало. Благодаря гранулярным локам эта проблема полностью решена.

4. **Очистка структуры Engine:**
   - Значительно упрощена функция `manageResources` и методы расчета скоростей.
   - Все неиспользуемые переменные и избыточная логика "эпизодов деградации" удалены, делая код `engine.go` более идиоматичным (idiomatic Go), читаемым и безопасным.
