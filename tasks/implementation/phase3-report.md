# Отчет по реализации: Фаза 3 (Streaming Engine)

## Что было сделано:
1. Создан модуль `internal/engine/stream.go` со структурой `StreamManager`, которая отвечает за счетчик ссылок (Reference Counting) стримов на каждый отдельный файл торрента.
2. В `StreamManager` реализован метод `AddStream`, который:
   - Увеличивает счетчик `ActiveStreams`.
   - Если счетчик стал равен 1, то для файла устанавливается наивысший приоритет (`PiecePriorityHigh`), а для остальных файлов в раздаче приоритет понижается до нормального (режим "Эгоиста").
   - Принимает `context.Context` (HTTP-запроса) и автоматически вызывает `RemoveStream` при его отмене.
3. Реализован механизм Debounce в методе `RemoveStream`:
   - Когда счетчик опускается до 0, запускается `time.Timer` на 10 секунд (`DebounceDelay`).
   - Если за это время поступает новый запрос (например, плеер сделал Seek-перемотку), таймер отменяется.
   - Если таймер истекает, снимается режим Sequential, и все файлы торрента возвращаются к нормальному приоритету (rarest-first фоновая загрузка).
4. Написана логика эмуляции кэша (`internal/engine/cache_emul.go`):
   - Метод `GetCacheStatus` вычисляет непрерывное "скользящее окно" загруженных `Piece` вперед от текущего индекса воспроизведения (offset).
   - Внедрено жесткое ограничение `virtualCacheBytes` на базе конфига (`HUB_STREAM_CACHE_SIZE`).
   - Если файл уже скачан на 100%, алгоритм мгновенно возвращает оставшийся размер файла.
5. Написаны и запущены юнит-тесты:
   - `internal/engine/stream_test.go`: проверка поведения контекста, отмены горутин и Debounce-счетчика, а также перехват ошибок на несуществующих торрентах.
   - `internal/engine/cache_emul_test.go`: юнит-тест эмуляции кэша с проверкой пограничных состояний на фиктивном (mock) движке.
