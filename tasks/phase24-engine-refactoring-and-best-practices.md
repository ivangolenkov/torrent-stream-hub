# Phase 24: Engine Refactoring & Best Practices

## Описание
Текущая кодовая база `internal/engine` стала слишком монолитной (God Object), использует опасные глобальные блокировки и переусложненные механизмы "самолечения" торрентов (HardRefresh, RecycleClient), что приводит к нестабильности и невозможности полноценного тестирования.
В данной фазе мы приведем архитектуру движка к стандартам idiomatic Go (Go Best Practices) без изменения внешнего поведения системы (API останется прежним).

## Задачи

### 0. Safety Net (Тестирование перед изменениями)
- [ ] Написать интеграционные тесты для `engine.go` (в файле `engine_test.go` или подобном), покрывающие жизненный цикл торрента (Add, Pause, Resume, Delete, ResourceMonitor).
- [ ] Убедиться, что текущие тесты (включая добавленные) проходят успешно, чтобы зафиксировать текущее поведение перед глубоким рефакторингом.

### 1. Ампутация переусложненного Swarm Watchdog
- [ ] Полностью удалить логику `HardRefresh` и `RecycleClient` из движка и API.
- [ ] Удалить структуры, отвечающие за сохранение состояния при пересоздании (`hardRefreshSavedState`, `recycleTorrentSnapshot`, логику дампа и восстановления метаданных).
- [ ] Оставить только легковесный `SoftRefresh` (повторный анонс трекерам/DHT, если скорость долго равна 0).
- [ ] Удалить ручной расчет скоростей в `engine.go` и перейти на использование стандартных метрик (`t.Stats().ConnStats.BytesReadData`).

### 2. Гранулярные блокировки (Fine-grained locking)
- [ ] Снять глобальную блокировку `Engine.mu` (RWMutex) со всех ресурсоемких операций и циклов. Глобальный лок должен защищать *только* операции добавления/удаления в коллекцию `managedTorrents`.
- [ ] Добавить локальный `mu sync.Mutex` в структуру `ManagedTorrent`. Все операции изменения состояния (Pause, Resume, подсчет статистики) должны блокировать только конкретный инстанс торрента.
- [ ] Устранить потенциальные дедлоки между `Engine.mu` и внутренними callback'ами `anacrolix/torrent`.

### 3. Декомпозиция Engine (Separation of Concerns)
- [ ] Вынести логику проверки свободного места на диске в отдельный независимый компонент (`DiskMonitor`).
- [ ] Вынести логику инициализации клиента и сетевых проверок (IP discovery) в конфигурационный этап или отдельный фабричный компонент.
- [ ] Четко изолировать `StreamManager`, чтобы он общался с движком без перекрестных блокировок (избавиться от сложного жонглирования мьютексами в `setSequentialMode`).

## Definition of Done (DoD)
- [ ] Добавлены интеграционные тесты для жизненного цикла торрента в движке (Add, Pause, Resume, etc.).
- [ ] Убраны `HardRefresh` и `RecycleClient` из всего проекта.
- [ ] Структура `Engine` не имеет глобального лока на время обхода всех торрентов; каждый `ManagedTorrent` имеет свой собственный лок.
- [ ] Логика работы с диском вынесена в `DiskMonitor`.
- [ ] Проект успешно компилируется (`go build ./cmd/hub`).
- [ ] Все тесты (`go test ./...`) проходят, показывая, что базовое поведение клиента не сломано.
- [ ] Отчет по рефакторингу составлен в `tasks/implementation/phase24-report.md`.
