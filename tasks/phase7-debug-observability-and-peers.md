# Фаза 7: Отладочная наблюдаемость и информация о пирах

Подготовить приложение к диагностике проблем с нулевой скоростью скачивания: добавить полезные debug-логи в критические точки torrent lifecycle и вывести информацию о пирах в Web GUI.

Цель фазы — не исправлять конкретную проблему скорости, а сделать состояние torrent engine, tracker/DHT discovery, peer connections и очереди загрузки достаточно прозрачными для локализации причины.

## Задачи

1. **Настраиваемое логирование:**
   - Ввести единый подход к логированию в backend: уровни `debug`, `info`, `warn`, `error`.
   - Добавить настройку уровня логов через ENV-переменную.
   - По умолчанию включить максимально подробный режим логирования, чтобы локальная диагностика работала без дополнительных настроек.
   - Использовать человекочитаемый текстовый формат логов, пригодный для просмотра через stdout/docker logs.
   - Не логировать секретные данные из magnet URL, включая passkey/private tracker query parameters; в логах использовать info hash, имя торрента и безопасно обрезанные значения.
   - Не добавлять хранение истории логов в памяти или отдельный UI для логов на этом этапе.

2. **Debug-логи torrent engine:**
   - Логировать добавление torrent-а из magnet, `.torrent` файла и bare info hash fallback.
   - Логировать получение metadata, изменение имени, списка файлов и общего размера.
   - Логировать переходы состояний: `Queued`, `Downloading`, `Paused`, `Seeding`, `Completed`, `Error`, `DiskFull`.
   - Логировать pause/resume/delete/restore, включая сценарии DB-only torrent и restore после рестарта.
   - Логировать решения resource manager: запуск из очереди, остановка из-за лимитов, нехватка места, активные загрузки.
   - Логировать tracker/DHT-related ошибки и важные события, если они доступны через `anacrolix/torrent` без чрезмерного шума.
   - Логировать скорость, прогресс и peer-сводку периодически или при значимых изменениях, с защитой от слишком частых сообщений.

3. **Debug-логи streaming/QoS:**
   - Логировать старт/завершение stream-сессии, `Range`-запросы и выбранный файл.
   - Логировать reference counting активных stream-ов.
   - Логировать включение/снятие sequential priorities и debounce при seek.
   - Логировать причины ошибок streaming: torrent не найден, metadata не готова, файл не найден, read/seek error.

4. **Peer-информация в backend API:**
   - Расширить backend-модель torrent-а или отдельный DTO данными о пирах.
   - Отдавать только агрегированные счетчики: количество известных peer-ов, количество подключенных peer-ов, seed/leech counts если доступны, количество активных соединений, tracker/DHT статус если доступен.
   - Не раскрывать IP-адреса, порты или другие per-peer details в Web GUI/API на этом этапе.
   - Обеспечить корректные значения до получения metadata и для DB-only/restored torrent-ов.
   - Сохранение peer-информации в SQLite не требуется, если это runtime-состояние engine.

5. **Peer-информация в Web GUI:**
   - Добавить отображение peer-сводки в таблицу torrent-ов или инспектор torrent-а.
   - Показать понятное состояние, когда metadata еще не получена или peer-информация недоступна.
   - Обновлять peer-информацию через существующий SSE/refresh flow.
   - Не перегружать главный экран детальными техническими полями; подробности вынести в инспектор при необходимости.

6. **Тесты и проверка:**
   - Добавить unit-тесты для форматирования/редакции безопасных log-полей, если появится отдельная логика sanitization.
   - Обновить backend API tests для новых peer-полей.
   - Обновить frontend types и проверить сборку.
   - Убедиться, что debug-логи не ломают существующие сценарии и не меняют публичное поведение API кроме добавления новых полей.

7. **Отчет по фазе:**
   - Создать `tasks/implementation/phase7-report.md`.
   - Описать добавленные точки логирования, новые peer-поля и способ включения debug-режима.
   - Зафиксировать ограничения: какие tracker/DHT/peer данные недоступны напрямую и чем это компенсировано.

## Definition of Done (DoD)

- [ ] У backend есть настраиваемый уровень логирования, включая `debug`.
- [ ] Уровень логирования настраивается через ENV-переменную, а по умолчанию включен максимально подробный режим.
- [ ] Логи выводятся в человекочитаемом текстовом формате через stdout/docker logs.
- [ ] Debug-логи покрывают add/restore/pause/resume/delete, metadata lifecycle, resource manager, streaming/QoS и основные ошибки.
- [ ] Логи не раскрывают passkey или другие чувствительные параметры magnet/private tracker URL.
- [ ] Backend API возвращает агрегированную runtime-сводку по peer-ам для каждого torrent-а без IP/port/per-peer details.
- [ ] Web GUI отображает peer-информацию и корректно обрабатывает состояния `metadata pending`/`unknown`.
- [ ] SSE или существующий refresh flow обновляет peer-информацию без ручной перезагрузки страницы.
- [ ] Добавлены или обновлены релевантные тесты.
- [ ] Успешно проходят `go test ./...`, `npm run build`, `CGO_ENABLED=0 go build ./cmd/hub`.
- [ ] Создан отчет `tasks/implementation/phase7-report.md`.

## Принятые решения

- Peer-информация ограничивается агрегированными счетчиками; подробные IP/port peer-ов не показываются.
- Формат логов — человекочитаемый текст, не JSON lines.
- Логирование настраивается только через ENV-переменную при старте приложения; runtime-переключатель в Web GUI не нужен.
- История debug-событий в памяти и отдельный UI для логов не нужны; достаточно stdout/docker logs.
